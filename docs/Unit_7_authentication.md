This guide expands on those foundations. We will explore how to customize the user model, build alternative login methods, manage granular permissions, and understand the core mechanics of web sessions.

## 1. Customizing the User Model

Django provides a built-in User model containing essential fields like Username, Email, and Password. However, most real-world applications require more information about a user. There are two primary ways to approach this.

**The Profile Approach (One-to-One)**
You can create a separate model that holds additional fields (like a bio or a profile picture) and link it to the built-in User model using a `OneToOneField`. This is excellent for keeping authentication data strictly separated from application-specific user data.

```python
from django.db import models
from django.contrib.auth.models import User

class Profile(models.Model):
    user = models.OneToOneField(User, on_delete=models.CASCADE)
    bio = models.TextField(max_length=500, blank=True)
    image = models.ImageField(default='default.jpg', upload_to='profile_pics')

```

**Extending AbstractUser**
If you know from the very beginning of your project that you need to fundamentally change how the user object behaves, you can create a custom user model by inheriting from `AbstractUser`. This replaces Django's default user model entirely. Note that this must be done before you run your first database migration.

```python
from django.contrib.auth.models import AbstractUser
from django.db import models

class CustomUser(AbstractUser):
    company_name = models.CharField(max_length=100)
    is_premium_member = models.BooleanField(default=False)

```

## 2. Authentication Backends

By default, Django uses the `ModelBackend`, which authenticates users against the standard User model using their username and password.

Modern web applications frequently allow users to log in using either their username or their email address. Custom authentication backends can be created to support these alternative authentication methods. You achieve this by writing a custom class that overrides Django's default `authenticate` method.

```python
from django.contrib.auth.backends import ModelBackend
from django.contrib.auth import get_user_model

class EmailBackend(ModelBackend):
    def authenticate(self, request, username=None, password=None, **kwargs):
        UserModel = get_user_model()
        try:
            user = UserModel.objects.get(email=username)
        except UserModel.DoesNotExist:
            return None
        
        if user.check_password(password):
            return user
        return None

```

To activate this, you add the path of your custom class to an `AUTHENTICATION_BACKENDS` list in your `settings.py` file.

## 3. Securing Views with Permissions

You are already familiar with using `@login_required` to ensure only authenticated users can access a view. However, authentication only verifies *who* the user is. Authorization verifies *what* they are allowed to do.

The `@permission_required` decorator restricts access based on specific user permissions. Django automatically creates permissions for every model in your database (e.g., add, change, delete, view).

```python
from django.contrib.auth.decorators import permission_required
from django.shortcuts import render

@permission_required('blog.add_post', raise_exception=True)
def create_post(request):
    return render(request, 'blog/create_post.html')

```

You can assign permissions to users individually, but it is much more efficient to use Django's built-in **Groups**. You can create a group called "Editors" in the admin panel, assign the `add_post` and `change_post` permissions to that group, and then simply assign users to the "Editors" group.

## 4. Security Essentials

**Cross-Site Request Forgery (CSRF)**
Whenever you build a form in Django that uses the `POST` method, you must include the `{% csrf_token %}` template tag. This token is a unique, secret value generated by your server. When the form is submitted, Django checks this token. This ensures that the form submission originated from your actual website and not from a malicious third-party site attempting to trick the user's browser.

**Password Hashing**
Django never stores plain-text passwords in the database. When a user creates an account, Django runs their password through a cryptographic hashing algorithm (PBKDF2 by default) along with a random string of characters called a "salt". The resulting hash is what gets saved. Even if a database is compromised, the original passwords remain secure because hashing is a one-way mathematical function.

**The `next` Parameter**
When an unauthenticated user tries to access a protected page (like their profile), Django redirects them to the login page and appends a parameter to the URL: `?next=/profile/`. Once the user logs in successfully, Django reads this parameter and redirects them to their original intended destination rather than the default home page. Django includes strict internal validations to ensure this `next` parameter only redirects to secure, internal pages, preventing attackers from using it to bounce users to external phishing sites.

## 5. Under the Hood: How Sessions Work

HTTP is a "stateless" protocol. This means the server treats every single request as completely independent. It has no memory of what happened a second ago. So, how does the server remember that you are logged in as you navigate from page to page? The answer is **Sessions and Cookies**.

Here is exactly what happens when you submit the login form with the correct credentials:

1. **The Request:** The browser sends an HTTP POST request to the server containing your username, password, and the CSRF token.
2. **The Verification:** Django's authentication backend retrieves the user from the database. It hashes the password you just typed and compares it to the hash stored in the database.
3. **Session Creation:** Because the hashes match, Django creates a new "Session" in the database. It stores relevant data (like your User ID) and generates a unique, random, highly secure string called a `session_key`.
4. **The Cookie Delivery:** Django prepares the HTTP response to send back to your browser. Inside the headers of this response, it includes a `Set-Cookie` instruction. This specific cookie is named `sessionid` and contains the unique `session_key`.
5. **Subsequent Requests:** As you click around the website, your browser automatically attaches that `sessionid` cookie to every new HTTP request. Django intercepts the request, reads the cookie, looks up that specific key in the database, identifies your User ID, and successfully authenticates your request.

### Hands-On: Exploring Sessions

You can observe this process happening in real-time.

**Explore the Browser:**

* Start your Django development server and log in to your application.
* Open your browser's Developer Tools (Right-Click -> Inspect, or press F12).
* Navigate to the **Application** tab (in Chrome/Edge) or the **Storage** tab (in Firefox).
* Expand the **Cookies** section in the left sidebar and click on `http://127.0.0.1:8000`.
* Look for the row with the Name `sessionid`. The Value is your current session key.
* *Experiment:* Right-click and delete that cookie, then refresh your page. You will be instantly logged out because you threw away your identifying "name tag."

**Explore the Database:**

* Open your project's SQLite database using a tool like DB Browser for SQLite.
* Browse the data inside the `django_session` table.
* You will see a `session_key` column. If you log back into your app, you will find a key here that perfectly matches the `sessionid` value in your browser's Developer Tools.
* The `session_data` column contains your encoded User ID, and the `expire_date` determines when you will be automatically logged out by the server.
